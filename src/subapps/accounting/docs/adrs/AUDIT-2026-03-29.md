## A. BUGS — Πρέπει να διορθωθούν

### A-1. ΚΡΙΣΙΜΟ: Ακύρωση τιμολογίου δεν αναστρέφει journal entry

**Σοβαρότητα**: 🔴 ΚΡΙΣΙΜΟ
**Αρχεία**:
- `src/app/api/accounting/invoices/[id]/route.ts` (DELETE handler)
- `src/subapps/accounting/services/accounting-service.ts`

**Πρόβλημα**: Όταν ακυρώνεται ένα τιμολόγιο (soft delete → `mydata.status = 'cancelled'`), η αυτόματη εγγραφή ημερολογίου που δημιουργήθηκε από `createJournalEntryFromInvoice()` **παραμένει ενεργή**.

**Αντίκτυπος**:
- Πλασματικά έσοδα στο ημερολόγιο
- Εσφαλμένος υπολογισμός ΦΠΑ (VAT Engine διαβάζει journal entries)
- Εσφαλμένος υπολογισμός φόρου εισοδήματος
- Εσφαλμένες τριμηνιαίες/ετήσιες αναφορές

**Απόφαση (2026-03-29)**: ✅ **Αντιλογιστική εγγραφή (Reversal Entry)** — Industry standard (SAP, Oracle, QuickBooks, Xero, SAGE).

**Λύση**: Δημιουργία μεθόδου `reverseJournalEntryForCancelledInvoice()` στο `AccountingService` που:
1. Βρίσκει τη journal entry με `linkedInvoiceId === invoiceId`
2. Δημιουργεί **νέα** journal entry με αντίθετα ποσά + `status='reversal'` + `reversedEntryId`
3. Η αρχική εγγραφή **ΠΟΤΕ δεν αλλάζει** (fiscal immutability)
4. Καλείται αυτόματα από τον DELETE handler
5. Reports αυτομάτως σωστά (αρχικό + reversal = 0)

**Εμπλουτισμός μετά από Industry Research (2026-03-29):**

6. **MANDATORY Reason Field** — Απόφαση: **Predefined + free text** (SAP/Oracle pattern, ΑΑΔΕ compliance)
   - Dropdown με 6 προκαθορισμένους λόγους:
     - `BILLING_ERROR` — Λάθος ποσό / τιμολόγηση
     - `DUPLICATE` — Διπλότυπο παραστατικό
     - `ORDER_CANCELLED` — Ακύρωση παραγγελίας
     - `TERMS_CHANGED` — Αλλαγή όρων
     - `GOODS_RETURNED` — Επιστροφή προϊόντων
     - `OTHER` — Άλλο (υποχρεωτικό free text)
   - Πεδίο `cancellationReason: CancellationReasonCode` (required)
   - Πεδίο `cancellationNotes: string` (required όταν reason='OTHER')
   - **ΑΑΔΕ**: Η αιτιολογία αποστέλλεται στο ακυρωτικό παραστατικό myDATA

7. **Ελληνική Νομοθεσία — Διαχωρισμός Void vs Credit Note** — Απόφαση: **Και τα δύο στη Φάση 1**
   - **Draft τιμολόγια** (δεν έχουν εκδοθεί):
     - Επιτρέπεται **void** (ακύρωση) — soft delete + journal reversal
     - Δεν χρειάζεται πιστωτικό (δεν εκδόθηκε ποτέ)
   - **Εκδοθέντα τιμολόγια** (sent / approved / paid):
     - **ΑΠΑΓΟΡΕΥΕΤΑΙ** void — ελληνικός νόμος
     - **ΜΟΝΟ πιστωτικό τιμολόγιο** (credit note)
     - Νέο document τύπου `credit_note` με αναφορά στο αρχικό τιμολόγιο
     - Journal reversal entry (αντιλογιστική εγγραφή) μέσω credit note
     - **Paid τιμολόγια**: Credit note + refund process
   - **API Logic**:
     - DELETE handler ελέγχει invoice status
     - `status === 'draft'` → void + journal reversal
     - `status === 'sent' | 'approved' | 'paid'` → δημιουργία credit note + journal reversal
     - **BLOCK**: Δεν επιτρέπεται απλή ακύρωση σε εκδοθέν τιμολόγιο
   - **ΑΑΔΕ/myDATA**: Το πιστωτικό τιμολόγιο υποβάλλεται ως ξεχωριστό παραστατικό

8. **Bidirectional Linking + Lock** — Απόφαση: **Πλήρες** (SAP/Oracle pattern)
   - Αρχική journal entry: `status → 'REVERSED'`, `reversalEntryId → <new entry ID>`, `reversedAt`, `reversedBy`
   - Reversal entry: `originalEntryId → <original ID>`, `originalEntryNumber`
   - **Immutability**: Μετά το reversal, ΚΑΝΕΝΑ από τα δύο entries δεν μπορεί να τροποποιηθεί
   - API enforce: PATCH σε reversed/reversal entries → 403 Forbidden

9. **Cascading Updates — Separate Balances** — Απόφαση: **Enterprise** (SAP/Oracle pattern)
   - Δημιουργία `customer_balances` collection (ή embedded document) που ενημερώνεται σε κάθε:
     - Νέο τιμολόγιο → αύξηση balance
     - Πληρωμή → μείωση balance
     - Credit note / reversal → μείωση balance
   - Πεδία: `customerId`, `totalOwed`, `totalPaid`, `netBalance`, `lastUpdated`
   - VAT liability ενημερώνεται αυτόματα μέσω journal reversal (ήδη on-the-fly)
   - Aging buckets: current, 30d, 60d, 90d, 120d+ (SAP standard)
   - **Ατομική ενημέρωση**: Firestore transaction — invoice cancel + journal reversal + balance update σε 1 atomic operation

10. **Full Fiscal Period Management** — Απόφαση: **Enterprise** (SAP/Oracle pattern)
    - Νέα collection `fiscal_periods` με:
      - `periodId`: `2026-Q1`, `2026-Q2`, κλπ (τριμηνιαία για ΦΠΑ)
      - `status`: `open` | `closed` | `locked`
      - `startDate`, `endDate`
      - `closedAt`, `closedBy` (audit)
    - **Κανόνες**:
      - Εγγραφή σε `open` period → OK
      - Εγγραφή σε `closed` period → BLOCK (403 Forbidden)
      - Reversal αρχικής εγγραφής σε κλειστή περίοδο → reversal στην τρέχουσα ανοιχτή + flag `differentPeriod: true`
      - Admin override: μπορεί να ξανανοίξει κλειστή περίοδο
    - **VAT alignment**: Οι περίοδοι ευθυγραμμίζονται με τις τριμηνιαίες δηλώσεις ΦΠΑ (Ελλάδα)
    - **Dashboard**: Νέο section "Φορολογικές Περίοδοι" στο Accounting Dashboard

11. **Ξεχωριστό Accounting Audit Log** — Απόφαση: **Enterprise** (SAP/Oracle pattern)
    - Νέα collection `accounting_audit_log` (ξεχωριστή από generic audit)
    - Πεδία:
      - `action`: `INVOICE_VOID` | `CREDIT_NOTE_ISSUED` | `JOURNAL_REVERSAL` | `PERIOD_CLOSED` | `BALANCE_ADJUSTED`
      - `documentId`, `documentType`, `documentNumber`
      - `amount`, `currency`
      - `reason`, `reasonCode`
      - `previousState`, `newState`
      - `fiscalPeriod`
      - `userId`, `userName`, `ipAddress`
      - `timestamp` (immutable)
    - **Immutable**: Κανένα UPDATE/DELETE σε audit entries
    - **Retention**: Minimum 10 χρόνια (ελληνική φορολογική νομοθεσία)

**Εκτίμηση εργασίας**: ~8 ώρες (σημαντική αύξηση λόγω credit note + balances + period management + audit log + ελληνικής συμμόρφωσης)

### A-5. Bank Match API χωρίς Zod Validation

**Σοβαρότητα**: 🟡 ΜΕΤΡΙΑ (ασφάλεια)
**Αρχείο**: `src/app/api/accounting/bank/match/route.ts:47`

**Πρόβλημα**: `const body = (await req.json()) as MatchRequestBody` — direct cast χωρίς Zod validation. Όλα τα υπόλοιπα endpoints χρησιμοποιούν `safeParseBody()`.

**Λύση**: Προσθήκη Zod schema + `safeParseBody()` pattern.

**Εμπλουτισμός μετά από Industry Research (2026-03-29):**

Πέρα από βασικό Zod, enterprise-grade validation (SAP/Xero/Sage pattern):
1. **Amount precision**: `z.number().positive().multipleOf(0.01).max(999_999_999.99)` — max 2 decimals, positive, finite
2. **Already-reconciled check**: Server-side ελέγχει αν η bank transaction ή το invoice είναι ήδη matched → reject
3. **Amount re-verification server-side**: Δεν εμπιστευόμαστε τα ποσά από το client — re-fetch από Firestore
4. **Status machine**: Bank transaction ακολουθεί strict states: `imported` → `matched` → `reconciled`. Δεν επιτρέπεται skip.
5. **Duplicate match prevention**: Αν η ίδια transaction+invoice combo υπάρχει ήδη → reject
6. **Structured error responses**: Field-level error codes (π.χ. `ALREADY_RECONCILED`, `AMOUNT_MISMATCH`, `DUPLICATE_MATCH`)

**Εκτίμηση εργασίας**: ~1 ώρα (αυξημένη λόγω enterprise validations)

**Αποφάσεις Γιώργου (2026-03-30) — Πρώτος γύρος:**

| # | Ερώτηση | Απόφαση |
|---|---------|---------|
| Q1 | Status machine: προσθήκη `reconciled` κατάστασης μετά το `matched`; | ✅ ΝΑΙ — 5η κατάσταση `reconciled` (κλειδωμένη, δεν αλλάζει). Flow: `unmatched` → `auto_matched`/`manual_matched` → `reconciled`. Χρειάζεται ξεχωριστό βήμα επιβεβαίωσης. |
| Q2 | Amount tolerance: exact match ή ±tolerance; | ✅ Enterprise SAP pattern — ±0.05€ tolerance. Διαφορά εντός tolerance καταχωρείται αυτόματα σε λογαριασμό στρογγυλοποιήσεων (54.99). Server-side re-verification πάντα. Configurable ανά εταιρεία. |
| Q3 | Duplicate match: μπλοκάρισμα ή αντικατάσταση; | ✅ Μπλοκάρισμα (SAP/Oracle pattern) — Αν η συναλλαγή είναι ήδη matched, reject με μήνυμα. Ο χρήστης πρέπει πρώτα να κάνει unmatch και μετά νέο match. Πιο ασφαλές, αποτρέπει ατυχήματα. |
| Q4 | Ξεκλείδωμα reconciled: αμετάκλητο, admin-only, ή ελεύθερο; | ✅ Admin-only με λόγο (SAP pattern) — Μόνο admin/λογιστής, υποχρεωτικός λόγος (dropdown), audit log entry. Η συναλλαγή γυρνά σε `matched` (όχι `unmatched`). Ξεχωριστό unmatch αν χρειάζεται νέο match. |
| Q5 | Error responses: απλό κείμενο ή structured; | ✅ Structured errors (enterprise standard) — Κωδικός (`ALREADY_MATCHED`, `AMOUNT_MISMATCH`, `PERIOD_LOCKED`, κλπ) + μήνυμα + field. Επιτρέπει field-level UI highlighting, i18n, programmatic handling. |
| Q6 | Fiscal period check: hard block, warning, ή skip; | ✅ Hard block (SAP pattern) — Αν η περίοδος είναι LOCKED/CLOSED, απαγορεύεται match. Ο χρήστης πρέπει πρώτα reopen (admin) → match → re-lock. Λόγος: ΦΠΑ/ΑΑΔΕ — αλλαγές σε κλειδωμένη περίοδο = φορολογικό πρόβλημα. |
| Q7 | Concurrency: optimistic locking με version number; | ✅ ΝΑΙ — Optimistic locking + Firestore transaction. Conflict error δείχνει **ΠΟΙΟΣ** τροποποίησε (όνομα χρήστη + πότε) + αυτόματο refresh σελίδας. Κάθε match αποθηκεύει `matchedByName` για conflict UX. |
| Q8 | Segregation of duties: matcher ≠ reconciler; | ✅ ΝΑΙ — Ο χρήστης που κάνει match δεν μπορεί να κάνει reconcile στην ίδια συναλλαγή. **Εξαίρεση: Super Admin bypass** — ο super admin μπορεί να κάνει τα πάντα. Configurable ανά εταιρεία (OFF για ατομικές με 1 άτομο). |
| Q9 | Refunds: match αρνητικών ποσών τώρα ή αργότερα; | ✅ ΤΩΡΑ — Υποστήριξη match αρνητικών ποσών (refunds/επιστροφές). Ο αλγόριθμος ελέγχει direction matching: πίστωση (credit) → κανονικό τιμολόγιο, χρέωση (debit) → πιστωτικό τιμολόγιο (credit note). Mismatch → reject με `BANK_MATCH_004 DIRECTION_MISMATCH`. |
