# ADR-268: Dynamic Report Builder

**Status**: DRAFT
**Date**: 2026-03-29
**Author**: Claude (για Γιώργο Παγώνη)
**Related ADRs**: ADR-265 (Enterprise Reports), ADR-266 (Gantt Reports), UC-007 (On-Demand Reports), UC-008 (AI Dashboards)

---

## 1. Πρόβλημα

Η εφαρμογή έχει 9 hardcoded report pages (ADR-265) αλλά χρειάζονται **εκατοντάδες έως χιλιάδες** ad-hoc αναφορές. Δεν μπορούμε να φτιάξουμε ξεχωριστή σελίδα για κάθε μία.

## 2. Απόφαση

Δημιουργούμε **1 δυναμικό Report Builder** στο `/reports/builder` που παράγει χιλιάδες αναφορές από **20+ domains** με δυναμικά filters, grouping, aggregations, και export.

## 3. Αρχιτεκτονικές Αποφάσεις

| # | Απόφαση | Σκεπτικό |
|---|---------|----------|
| D1 | Server-side queries (API route) | Security + performance |
| D2 | Reuse ADR-265 export infrastructure | Zero duplication |
| D3 | 20+ domain configs (not generic query) | Type safety, simplicity |
| D4 | Persona-based domains (Engineers, Workers, etc.) | Match business mental model |
| D5 | Max 2 group-by levels | Firestore/UX limits |
| D6 | Saved reports σε Firestore | Cross-device persistence |
| D7 | No scheduled reports (Phase 1) | Vercel Hobby limits |
| D8 | 500 row default limit | Performance guardrail |
| D9 | Column checkboxes (not drag-and-drop) | Sufficient, simpler |
| D10 | 3-tier export approach | Industry standard (Salesforce/Procore/Dynamics 365) |

## 4. 3-Tier Export Architecture

| Tier | Τι | Πώς | Pattern |
|------|-----|------|---------|
| **Tier 1** | Flat table — 1 row/record, primary values μόνο | Report Builder table + PDF/Excel | Universal standard |
| **Tier 2** | Row repetition — 1 row per child record | "Αναλυτική εξαγωγή" mode | Salesforce/SAP pattern |
| **Tier 3** | Contact Card PDF — nested formatted κάρτα | "Εκτύπωση Καρτέλας" button | Procore/Dynamics 365 pattern |

## 5. Σχέση με ADR-265

| ADR-265 (Existing) | ADR-268 (Report Builder) |
|-------------------|--------------------------|
| Predefined dashboards ανά domain | Ad-hoc queries, custom αναφορές |
| Quick overview, KPIs, trends | Specific business questions |
| Fixed layout, charts + tables | Configurable columns, filters, groups |
| `/reports/*` (9 pages) | `/reports/builder` (1 page) |

Τα δύο **συμπληρώνουν** το ένα το άλλο.

## 5.1 Σχέση με ADR-266 (Gantt Construction Reports)

| ADR-266 (Existing) | ADR-268 (Report Builder) |
|-------------------|--------------------------|
| Per-building construction analytics | Cross-building/cross-project queries |
| Specialized visualizations (S-Curve PV/EV/AC, CPM, Resource Histogram, Delay Breakdown) | Generic bar/pie charts |
| Tab μέσα στο Building Management | `/reports/builder` → domains D1-D4 |

**Διαχωρισμός ευθυνών**: Τα specialized construction visualizations (S-Curve, Critical Path Method, Resource Histogram) **παραμένουν αποκλειστικά** στο ADR-266. Ο Report Builder παρέχει μόνο generic tabular/chart queries cross-building (π.χ. "όλες οι καθυστερημένες φάσεις σε όλα τα κτίρια").

## 5.2 Σχέση με ADR-267 (Procurement Module)

| ADR-267 (Existing) | ADR-268 (Report Builder) |
|-------------------|--------------------------|
| Full CRUD module (workflow, approvals, delivery) | Read-only queries & export |
| 7 operational KPIs, Budget Overview charts | Dynamic aggregations (COUNT, SUM, AVG) |
| `/procurement` (standalone, top-level) | `/reports/builder` → domain C6 "Παραγγελίες" |

**Διαχωρισμός ευθυνών**: Operational actions (Create PO, Approve, Cancel, Record Delivery, Email, Share) **παραμένουν αποκλειστικά** στο ADR-267. Ο Report Builder είναι read-only analytics layer (π.χ. "ποια POs ξεπέρασαν τον budget cross-project;").

## 6. ΑΔΙΑΠΡΑΓΜΑΤΕΥΤΟ: SSoT — Centralized Systems Mapping

> **Κανόνας Γιώργου**: ΜΗΔΕΝ διπλότυπα, ΜΗΔΕΝ διάσπαρτος κώδικας, ΜΗΔΕΝ μη-Google-level κώδικας.
> Κάθε πτυχή του Report Builder ΠΡΕΠΕΙ να χρησιμοποιεί τα υπάρχοντα centralized systems.
> **Single Source of Truth (SSoT)** — κανένα νέο σύστημα αν υπάρχει ήδη ένα.

| **Report Core** | ADR-265 components | `src/components/reports/core/` | Νέο ReportTable, ReportChart, ReportKPIGrid |

### 6.6 Verification Checklist (Pre-Implementation)

Πριν γράψεις ΜΙΑ γραμμή κώδικα για το Report Builder:

- [ ] Grep για existing implementation πριν δημιουργήσεις νέο component
- [ ] Έλεγξε `docs/centralized-systems/README.md` για σχετικό σύστημα
- [ ] Έλεγξε `docs/centralized-systems/design-system/hooks.md` πριν γράψεις νέο hook
- [ ] Χρησιμοποίησε `src/config/firestore-collections.ts` (ΠΟΤΕ hardcoded collection names)
- [ ] Χρησιμοποίησε `enterprise-id.service.ts` για νέα document IDs
- [ ] Χρησιμοποίησε Radix Select (ΠΟΤΕ νέο dropdown)
- [ ] Extend existing i18n namespace (ΠΟΤΕ hardcoded strings)
- [ ] Reuse ADR-265 export functions (ΠΟΤΕ νέο PDF/Excel wrapper)

## 7. Τεχνικές Αποφάσεις

- **Βιβλιοθήκες**: Καμία νέα — reuse recharts, jspdf, exceljs, html-to-image
- **Data Layer**: Server-side aggregation via API route (Firestore Admin SDK)
- **UI**: Reuse ADR-265 core components (ReportTable, ReportChart, ReportKPIGrid)
- **i18n**: Extend reports namespace
- **68 Firestore collections** χαρτογραφήθηκαν (βλ. RESEARCH.md)
