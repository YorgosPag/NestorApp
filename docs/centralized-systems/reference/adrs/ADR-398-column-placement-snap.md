## 1. Πλαίσιο

Το column tool (ADR-363, freehand single-click) ζωγραφίζει ghost κάτω από το σταυρόνημα και τοποθετεί κολώνα στο κλικ. Το snap κατά την τοποθέτηση τρέχει σε **2 σημεία** (mirror): τον decoupled `snap-scheduler` (move → ghost/indicator, ADR-040 Φ11) και τον `mouse-handler-up` (click → commit point). Προϋπήρχε το **Column Body Corner Projection Snap** (η would-be κολώνα προβάλλει τις γωνίες της σε στόχους· `findColumnDrawCornerSnap`).

## 2. Πρόβλημα (Giorgio 2026-06-19)

Όταν ο χρήστης τρέχει «Κολώνα» και περνά το σταυρόνημα **πάνω από δοκάρι**, θέλει:
1. Οπτική ένδειξη ότι κλικ εκεί θα τοποθετήσει την κολώνα **πάνω στο δοκάρι**.
2. Η κολώνα να τοποθετείται **στον άξονα** του δοκαριού — **κέντρο κολώνας ≡ κέντρο άξονα δοκαριού**.

## 3. Απόφαση

### 3.1 Beam-axis projection SSoT (γέμισμα κενού)
Η οικογένεια per-member axis projection που καταναλώνει ο `NearestSnapEngine` είχε `projectPointOnWallAxis` / `projectPointOnSlabEdge` / `projectPointOnOpeningOutline` — **όχι** δοκάρι. NEW SSoT `bim/beams/beam-axis-projection.ts` (**ακριβές mirror** του `projectPointOnWallAxis`): `projectPointOnBeamAxis(beam, cursor)` (clamped nearest foot· διαβάζει cached `geometry.axisPolyline.points` → straight/curved uniform· ίδιες shared utils `getNearestPointOnLine`+`calculateDistance`) + `projectPointOnBeamAxisDetailed` (foot + distance + `atEndpoint`). Wired **`isBeamEntity` branch στο `NearestSnapEngine`** + description key `bim-beam` (`snapModes.labels.bim.beamAxis`, el+en) ⇒ **τα δοκάρια αποκτούν nearest-snap parity** με walls/slabs σε **ΟΛΑ** τα drawing tools. _(PerpendicularSnapEngine beam-feet = DEFER — εύκολη μελλοντική parity.)_

### 3.1b Column→Beam axis snap (κέντρο κολώνας ≡ άξονας δοκαριού)
NEW `bim/columns/column-placement-snap-context.ts` (column-tool orchestration **πάνω στο SSoT**, μηδέν inline geometry):
- `findColumnBeamAxisSnap(worldPos, beams)` — **reuse `projectPointOnBeamAxisDetailed`**· υποψήφιο όταν `atEndpoint===false` (πάνω στο σώμα, όχι πέρα από τα άκρα) ΚΑΙ `distance ≤ halfWidth · 1.5` (width-based ⇒ **transform-independent**)· νικά το μικρότερο.
- `resolveColumnPlacementContext(worldPos, entities)` → **precedence `overlap` > `beam` > `neutral`**: αν ο cursor είναι μέσα σε footprint **υπάρχουσας κολώνας** → `overlap` (🔴 — μην βάλεις διπλή· ισχύει **ΑΚΟΜΗ κι όταν από κάτω περνά δοκάρι**, π.χ. ενδιάμεση κολώνα στη μέση δοκαριού)· αλλιώς πάνω σε **κενό** σώμα δοκαριού → `beam` (🟢)· αλλιώς `neutral`. _(Το «δοκάρι νικά» αφορά τα snap **targets** grid/corner — όχι το collision warning έναντι υπάρχουσας κολώνας.)_

Wiring (mirror του corner-snap, ΕΝΑ SSoT σε 2 paths):
- **Move** (`snap-scheduler.runSnapDetection`): column active → `resolveColumnPlacementContext`· `beam` ⇒ `setImmediateSnap` στο σημείο του άξονα (+ beamId) και short-circuit του corner/center snap. `mouse-handler-move` τροφοδοτεί `getEntities`.
- **Commit** (`mouse-handler-up`): column active → beam-axis νικά → `worldPoint = beam point`.
- **Center enforcement** (`useColumnTool.commitColumnFromState`): όταν ghost status = `beam` → anchor **`'center'`** ανεξάρτητα από τον επιλεγμένο anchor ⇒ κέντρο κολώνας ≡ άξονας δοκού.

### 3.2 Ghost coloring (σημασιολογικό feedback στο ghost, ΟΧΙ στην υποκείμενη οντότητα)
Ο χρωματισμός γίνεται στο **ghost** (η προσοχή είναι εκεί· Revit/AutoCAD pattern· δεν αγγίζει τον select-gated entity-hover renderer):
- 🟢 **πράσινο** = πάνω σε δοκάρι (έγκυρος snap στόχος)
- 🔴 **κόκκινο** = πάνω σε υπάρχουσα κολώνα (σύγκρουση/επικάλυψη — τοποθετείται ακόμη, προειδοποίηση)
- ⚪ **default (χρώμα τύπου)** = ελεύθερη τοποθέτηση

NEW zero-React store `systems/cursor/ColumnPlacementGhostStatusStore.ts` (mirror `ImmediateSnapStore`): ο scheduler γράφει το status· το `useColumnGhostPreview` το διαβάζει **imperatively μέσα στο RAF** (zero subscription, ADR-040) και περνά `statusColor` στον `ColumnAnchorGhostRenderer` (νέο optional override, `resolveGhostStatusColor`).

### 3.3 Smart beam ghost — φάντασμα δοκαριού στις παρειές κολόνας (mirror, 2026-06-20)
Καθρέφτης του column ghost για το εργαλείο «Δοκάρι»: μόλις ενεργοποιηθεί, **πριν το 1ο κλικ** εμφανίζεται μικρό έξυπνο φάντασμα δοκαριού. Κοντά σε **ορθογώνια κολόνα** κουμπώνει σε μία από **12 διακριτές θέσεις** (4 παρειές × 3 αγκυρώσεις, το φάντασμα ΠΗΔΑΕΙ — όχι συνεχές γλίστρημα)· μακριά → ακολουθεί ελεύθερα τον κέρσορα.

**Κρίσιμη αρχιτεκτονική διαφορά από το column ghost:** η orientation/anchor είναι **καθαρά γεωμετρική** (από τα column footprints), ΟΧΙ από snap-result → **μηδέν επαφή με snap-scheduler / cursor stores / renderer core** (κανένα ADR-040 critical file, CHECK 6B/6C/6D). Όλη η λογική ζει στο **pure preview path** + στο click handler του tool.

- **Pure resolver** `bim/beams/beam-column-face-snap.ts` (`resolveBeamColumnFaceSnap` + wrapper `resolveBeamGhostSnapFromStore`): cursor + column footprints → `{ face:'E'|'W'|'N'|'S', third:'lo'|'mid'|'hi', start, end }` (centerline) ή `null`. **Reuse `projectPolygonOnAxis`** (extents, N.0.2) + `mmToSceneUnits`. Σημασιολογία: E/W → οριζόντιο δοκάρι, N/S → κάθετο· κοντινό short-end flush στην παρειά· `lo`/`hi` → γωνία-flush (πλάγια παρειά δοκού ≡ γωνιακή παρειά κολόνας), `mid` → centerline ≡ κέντρο κολόνας.
- **Centerline vs location-line:** ο resolver επιστρέφει το ΤΕΛΙΚΟ centerline `start`. Το 1ο κλικ κλειδώνει εκεί + σημαία `beamPreviewStore.startAnchored=true`· τότε το awaitingEnd preview/commit χρησιμοποιεί `buildDefaultBeamParams` (**centerline mode**), ΟΧΙ το location-line auto-flush (`buildAnchoredBeamParams`) που θα ξανα-μετατόπιζε το start ±width/2 σε διαγώνιο τράβηγμα. Free placement → αμετάβλητη location-line συμπεριφορά (ADR-363 §5.7).
- **Wiring (μηδέν duplicate, preview === commit):** ΕΝΑ wrapper καλείται και από `beam-preview-helpers` (ghost-before-click) και από `useBeamTool.onCanvasClick` (1ο κλικ). `syncColumnsToStore()` τρέχει πλέον και στο `activate` (κολόνες διαθέσιμες πριν το κλικ). **2ο κλικ ΕΛΕΥΘΕΡΟ** (διαγώνιο/ORTHO) — ο άξονας ΔΕΝ κλειδώνει.

### 3.4 Beam ghost +Y offset — canonical viewport SSoT στο WYSIWYG BIM preview (render-path, 2026-06-20)
**Πρόβλημα (preview-only):** το ghost δοκαριού (WYSIWYG, μέσω πραγματικού `BeamRenderer`) εμφανιζόταν με **σταθερό μικρό offset προς βορρά (+Y)** σε όλες τις παρειές/αγκυρώσεις — το committed δοκάρι ευθυγραμμιζόταν σωστά. Ομοιόμορφη μετατόπιση όλου του ghost, ανεξάρτητη zoom/πλάτους.

**Ρίζα (αιτιακή, όχι geometry):** ο y-flip είναι `screenY = (viewport.height − top) − worldY·scale − offsetY`. Ο beam ghost ζωγραφίζεται στο **shared PreviewCanvas** αλλά οι πραγματικοί entity renderers μετρούσαν το `viewport.height` από το **δικό τους** `ctx.canvas.getBoundingClientRect()` (= PreviewCanvas), ενώ **όλα τα σωστά** (committed δοκάρι, column ghost μέσω `getViewportElement()`) μετρούν από το **DxfCanvas / container rect** (= prop `viewport` = `containerRef.getBoundingClientRect`, `useViewportManager`). Το PreviewCanvas έχει inline `style.width/height` (από `PreviewRenderer.updateSize`) που αποκλίνει λίγα px από το container → σταθερό +Y. **Διπλοτυπία πηγής viewport:** δύο preview paths (imperative composite vs RAF-direct column ghost) έλυναν το «canonical viewport» με διαφορετικό τρόπο.

**Λύση (ενοποίηση σε ΕΝΑ canonical viewport — N.0.2):** NEW `setViewportOverride(viewport|null)` στο `BaseEntityRenderer` (όταν set → `getViewport()` το επιστρέφει αντί για getBoundingClientRect) + forwarding στο `EntityRendererComposite`. Το `BimPreviewRenderer.render(entity, transform, viewport)` εγχέει το canonical `viewport` (περασμένο από `PreviewRenderer` = prop viewport) πριν το render και το καθαρίζει σε `finally`. **Μηδέν regression:** override=`null` στο main render path → αμετάβλητη getBoundingClientRect συμπεριφορά. Τώρα ο beam ghost μετρά στο ΙΔΙΟ viewport με το committed δοκάρι + column ghost. Λύνει ΚΑΙ το «άξονας λίγα px πιο βόρεια από το σταυρόνημα» (ίδια αιτία).

### 3.5 Preview-render unification — ΕΝΑ canonical frame + ΕΝΑ ghost harness (2026-06-20)
Το §3.4 hotfix έλυσε το σύμπτωμα· το §3.5 εξαλείφει την **κλάση** του bug + τη **διπλοτυπία** που τον γέννησε (Revit-grade, ο Giorgio ζήτησε πλήρη ενοποίηση). Δύο SSoT, ΟΧΙ ένας mega-renderer (τα ghosts έχουν νόμιμα διαφορετική οπτική):

- **SSoT-A — canonical preview frame** (`systems/preview/ghost-preview-frame.ts`): `getCanonicalPreviewFrame(viewportElement)` → `{ viewport, transform }` από **ΜΙΑ** πηγή — viewport μέσω `canvasBoundsService.getBounds` (cached, ΙΔΙΑ πηγή με τον main `DxfRenderer`), transform μέσω live `getImmediateTransform()` (zero-lag, ΙΔΙΟ με main canvas + ProposalGhostOverlay). + uniform `GhostDrawFrame { ctx, effectiveCursor, viewport, transform }` + `GhostDrawDelegate`. Διορθώνει ΚΑΙ το **latent transform-lag**: μέχρι τώρα τα RAF-direct previews έπαιρναν React-prop transform (1-2 frames πίσω από τον main canvas σε pan/zoom).
- **SSoT-B — ghost harness** (`hooks/tools/useCanvasGhostPreview.ts`): κατέχει το copy-pasted scaffolding (~19 hooks) — cursor-gate, RAF lifecycle, DPR-clear, clear-on-exit, snapped-cursor, canonical frame. Config: `{ isActive, getCanvas, getViewportElement, transform, cursorMode?, useImmediateSnap?, clearMode?, draw }`. Modes: `cursorMode 'none'` (grip ghosts — cursor μέσω data), `clearMode 'skip-clear'` (`useGripDimAnnotation` — layered πάνω στο grip-ghost frame).
- **Migrated 19 hooks:** column/opening/slab-opening + 7 MEP (fixture/water-heater/radiator/electrical-panel/boiler/manifold/segment) + rotation/move/mirror/scale/stretch/trim/extend/grip-ghost/grip-dim. Κάθε hook → ΜΟΝΟ το `draw` delegate + το gate του (draw logic IDENTICAL, byte-for-byte οπτικά· ~600 γρ scaffolding διαγράφηκαν). `transform` prop μένει ως **redraw-trigger** (programmatic zoom/fit)· η ΖΩΓΡΑΦΙΖΟΜΕΝΗ τιμή = live transform.
- **ΟΧΙ migrated (by design):** ο imperative composite path (PreviewCanvas drawPreview — §3.4 viewport-override) + τα dedicated-canvas ProposalGhostOverlay (ήδη live transform). Διαφορετικός lifecycle.

### 3.6 Beam-to-beam framing — smart ghost ΚΑΙ πάνω σε υφιστάμενα δοκάρια (κατευθυντικό 🟢/🔴, 2026-06-20)
Επέκταση του §3.3 smart beam ghost: το φάντασμα κουμπώνει **και πάνω σε υφιστάμενα δοκάρια** (όχι μόνο κολόνες), με **κατευθυντικό χρωματισμό** (mirror του 🟢/🔴 column ghost §3.2). Δομικό σκεπτικό (Giorgio, πολιτικός μηχανικός): **Τ-junction** δευτερεύοντος σε μακριά παρειά κυρίου = standard beam-to-beam framing (η Revit το κάνει καθημερινά) → 🟢· **συγγραμμικά** άκρη-με-άκρη ή σώμα-σε-σώμα = θα έπρεπε να είναι ΕΝΑ δοκάρι → 🔴 (extend instead).

| Θέση cursor vs υφιστάμενο δοκάρι | Status | Συμπεριφορά ghost |
|---|---|---|
| **Μακριά παρειά** (Β/Ν) | 🟢 `beam` | κάθετο Τ-framing, ολίσθηση σε όλο το μήκος + μαγνήτες άκρα/μέσα |
| **Κέντρο άξονα** | 🟢 `beam` | auto-snap στην **πλησιέστερη** παρειά (απόφαση «Α» — ομαλό UX, ποτέ «καβάλα») |
| **Κοντή άκρη** (Α/Δ) | 🔴 `overlap` | συγγραμμική συνέχεια — «extend instead» (κόκκινο schematic ghost) |

**Κρίσιμη γεωμετρική διαφορά από το column ghost (§3.3):** τα δοκάρια είναι **μακρόστενα & πιθανώς υπό γωνία** → το ⊥/∥ κρίνεται από τον **άξονα** του υφιστάμενου δοκαριού, ΟΧΙ από axis-aligned bbox. Νέο **axis-relative** pure resolver, δεν «χώνεται» στον bbox-based column resolver.

**FULL SSoT (extract/generalize/reuse — μηδέν διπλότυπα):**
- **EXTRACT** η παλέτα 🟢/🔴 + ο `drawGhostOutline`/`drawGhostFill` βγήκαν από το (ADR-040 critical) `ColumnAnchorGhostRenderer` σε **ουδέτερα** `bim/ghosts/ghost-status-color.ts` (`GhostStatus`/`GhostStatusColor`/`resolveGhostStatusColor`) + `bim/ghosts/ghost-status-polygon-draw.ts` (`strokeGhostPolygon`/`fillGhostPolygon`/`drawStatusGhostPolygon`). Column renderer τα re-exports (back-compat `useColumnGhostPreview`) + delegate-άρει. **Beam→column-renderer coupling = 0.**
- **GENERALIZE** `findColumnOverlap` → ουδέτερο `bim/geometry/entity-overlap.ts` (`findEntityOverlap(pos, entities, getPolygon)` — extractor-driven, ίδιος point-in-polygon για κολόνα/footprint & δοκάρι/outline)· `findColumnOverlap` delegate.
- **REUSE** `projectPolygonOnAxis` (signed perp = μακριές παρειές + κοντές άκρες· straddle) + `projectPointOnAxis` (cursor along) — **μηδέν** νέο projection primitive.
- **NEW** `bim/beams/beam-beam-face-snap.ts` (`resolveBeamBeamFaceSnap` + `BeamSnapTarget`/`BeamGhostSnapResult`): cursor πάνω στο σώμα → 🟢 ⊥ Τ-framing στην πλησιέστερη παρειά (ολίσθηση κατά μήκος μέσω `tAlong`)· πέρα από κοντή άκρη → 🔴 συγγραμμική συνέχεια.

**Wiring (preview === commit, ΕΝΑ dispatcher):** ο `resolveBeamGhostSnapFromStore` έγινε **dispatcher** — **column-priority** (κοντά σε κολόνα η παρειά κολόνας νικά → status `neutral`, διατηρεί ΑΚΡΙΒΩΣ το §3.3 behavior)· αλλιώς beam-to-beam (🟢/🔴). `beamPreviewStore` απέκτησε `beamTargets` + `setBeams` (mirror `columnFootprints`/`setColumns`)· `useBeamTool.syncSceneTargetsToStore` (πρώην `syncColumnsToStore`) συγχρονίζει **και** δοκάρια (axis+outline) on activate / κάθε 1ο κλικ.

**Μαγνήτες άκρα/μέσα (Slice 2 — reuse, μηδέν νέος κώδικας):** το `makeBeamGhostBeforeClick` ήδη διαβάζει `getImmediateSnap()` (§3.4 fix)· η BIM characteristic snap (`BimCharacteristicSnapEngine`, ADR-370 — `BIM_CORNER`/`BIM_MIDPOINT` για **όλες** τις BIM οντότητες incl. δοκάρια) δίνει τα endpoint/midpoint· ο snapped cursor → το ghost κουμπώνει στο άκρο/μέσο.

**3-ζωνική δικαιολόγηση πλάτους (Slice 5, Giorgio §3.6):** το κάθετο ghost ΜΕΝΕΙ ίσιο (90°) — ΜΕΤΑΤΟΠΙΖΕΤΑΙ κατά τον άξονα u ώστε το σταυρόνημα να πέφτει στην **αρχή/μέση/τέλος του πλάτους** του, ανάλογα σε ποιο **τρίτο** του υφιστάμενου δοκαριού βρίσκεται (mirror των 3 αγκυρώσεων κολόνας). **ΒΟΡΕΙΑ:** αριστερό τρίτο → δοκάρι δεξιά (cursor=αριστερή ακμή, centerline `+half`)· μεσαίο → κεντραρισμένο· δεξί → δοκάρι αριστερά (`−half`). **ΝΟΤΙΑ: αντίστροφα.** **SSoT:** `pickThird` εξήχθη σε NEW zero-import leaf `bim/beams/beam-face-third.ts` (cycle-proof — το `beam-column-face-snap` εισάγει το `beam-beam-face-snap`)· **και** οι δύο resolvers το κάνουν reuse, μηδέν διπλότυπο. `beamWidthScene` περνά στο opts του resolver.

### 3.7 Column smart-ghost face-snap — η κολώνα κουμπώνει στις παρειές (πιστό mirror δοκαριού, 2026-06-20)

**Giorgio order:** το εργαλείο «Κολώνα» να αποκτήσει την ίδια «έξυπνη» συμπεριφορά φαντάσματος με το δοκάρι (§3.3/§3.6), **πιστά προσαρμοσμένη** σε σημειακό (1-κλικ) μέλος, με ΕΝΑ SSoT (reuse του `bim/framing/` ADR-508).

Κοντά σε παρειά υφιστάμενου **δοκαριού, τοίχου ή κολώνας** (ίδια συμπεριφορά — όλα γραμμικά/footprint μέλη), το φάντασμα της κολώνας:
- **κουμπώνει** στην παρειά και αναπτύσσεται **εξωτερικά** (το σώμα στην έξω πλευρά)·
- **γλιστράει** κατά μήκος της (continuous slide — η `position` ακολουθεί τον — ήδη snapped — cursor)·
- η ζώνη (lo/mid/hi via `pickThird`) επιλέγει **ΑΥΤΟΜΑΤΑ** ποια από τις 9 λαβές ακουμπά: γωνία (lo/hi) → flush-corner, μέσον (mid) → κεντραρισμένη — ο σημειακός ισοδύναμος της «ανάπτυξης Α/Δ» του δοκαριού·
- χρωματισμός: μακριά παρειά → 🟢 `beam` (valid attach)· **κοντή άκρη ΔΟΚΑΡΙΟΥ** → 🔴 `overlap` (mirror «extend instead»)· **κολώνα ΚΑΙ ΤΟΙΧΟΣ → ΟΛΕΣ οι παρειές 🟢** (Giorgio: σε κάθε πλευρά τοίχου, **και τις μικρές άκρες**, επιτρέπεται κολώνα — μόνο το δοκάρι έχει 🔴 κοντές άκρες)·
- **έλξεις/γλυφές (`resolveColumnFaceSnapWithGlyph`):** όταν ο cursor πλησιάζει **ορατό BIM χαρακτηριστικό** (Γωνία/Μέσο/Κέντρο τοίχου/δοκαριού/κολόνας — μέσω του ενιαίου `findSnapPoint`), το ghost κάνει **μαγνητική έλξη** εκεί ΚΑΙ εμφανίζεται η γλυφή + ετικέτα («ΓΩΝΙΑ ΤΟΙΧΟΥ»/«ΜΕΣΟ ΤΟΙΧΟΥ»)· μακριά από χαρακτηριστικό → ελεύθερο slide χωρίς γλυφή (σιωπηλό grid/guide αγνοείται, AutoCAD convention).

**Anchor mapping** (παρειά στόχου × ζώνη → λαβή· η απέναντι ακμή της κολώνας ακουμπά, σώμα εξωτερικά):
| Παρειά | `lo` | `mid` | `hi` |
|---|---|---|---|
| **N** (σώμα βόρεια) | `sw` | `s` | `se` |
| **S** (σώμα νότια) | `nw` | `n` | `ne` |
| **E** (σώμα ανατολικά) | `sw` | `w` | `nw` |
| **W** (σώμα δυτικά) | `se` | `e` | `ne` |

**FULL SSoT — μηδέν νέο engine/geometry, μηδέν διπλότυπο:**
- **EXTRACT** κοινό geometry SSoT `bim/geometry/shared/footprint-face-frame.ts` (`footprintBounds` + `distanceToFootprintBounds` + `pickDominantFace`) — το bbox/face primitive που ΠΡΙΝ ήταν **διπλό** (private `columnBounds`/`distanceToBounds` + inline ex/ey στο `member-column-face-snap` **και** ξανά στην 1η μου έκδοση column). Πλέον ζει ΜΙΑ φορά· **και τα δύο** (`member-column-face-snap` δοκάρι/τοίχος **ΚΑΙ** `column-face-snap` κολώνα) το καταναλώνουν → πραγματική μία πηγή αλήθειας (byte-for-byte: `footprintBounds` = το παλιό `columnBounds` μέσω `projectPolygonOnAxis`).
- NEW pure `bim/columns/column-face-snap.ts` (`resolveColumnFaceSnap`) — **reuse** `collectMemberSnapTargets` (στόχοι, `memberKinds:['beam','wall']` → κολόνες+δοκάρια+**τοίχοι**, ίδια συμπεριφορά φαντάσματος), το νέο `footprint-face-frame` SSoT, `pickThird` (zone), `MEMBER_GHOST_CAPTURE_MM` (ίδιο capture 600). Η τελική γεωμετρία (anchor offset) παράγεται από το υπάρχον `computeColumnGeometry` — **κανένα νέο geometry**.
- **ΕΝΑ SSoT για move ΚΑΙ click** (preview === commit): NEW `resolveColumnFaceSnapWithGlyph` (στο `column-placement-snap-context`) = `findSnapPoint` → ορατό χαρακτηριστικό ως effective cursor (μαγνητική έλξη) + `resolveColumnFaceSnap`. - `useColumnGhostPreview` draw: όταν `getColumnFaceAnchor()` set → ζωγραφίζει **ΜΟΝΟ** εκείνο το anchor ghost (active, στο face point), αλλιώς τα 9 (imperative read, ADR-040-safe).

**Σημείωση:** η νέα face-attach συμπεριφορά **υπερισχύει** της §3.1b «center on beam axis» όταν ο cursor είναι κοντά σε παρειά δοκαριού (τα δοκάρια είναι λεπτά → σχεδόν πάντα εντός capture) — εσκεμμένη αναβάθμιση κατά απαίτηση Giorgio. _(Circular column: το `computeColumnGeometry` αγνοεί anchor → κεντράρει στο face point· edge-case, follow-up.)_

### 3.9 Wall-axis CENTER snap — κέντρο κολώνας στον άξονα τοίχου (mirror §3.1b, 2026-06-21)

**Giorgio order (Revit-grade):** το εργαλείο «Κολώνα» πάνω σε **τοίχο** ΔΕΝ επέτρεπε τοποθέτηση **κεντραρισμένη στον άξονα** του τοίχου — το §3.7 face-snap κουμπώνει **μόνο στις παρειές** (flush). Ζητούμενο: ακριβές **mirror του §3.1b «Column→Beam axis»** (status → center-anchor), αλλά για τοίχο. **ΧΩΡΙΣ split τοίχου** (η «φάτνωμα ανά κολώνα» υποδιαίρεση = αναλυτικό μοντέλο organism/FEM, ΟΧΙ φυσικό element — ADR-487).

**Απόφαση (κλειδωμένη):** **nearest-reference-wins** μέσα στον τοίχο (όπως Revit «centerline + face references»):
- ο cursor **πιο κοντά στον άξονα** (εσωτερική μισή ζώνη, perp ≤ ½ ημι-πάχους) → **center-on-axis**: το ΚΕΝΤΡΟ της κολώνας κουμπώνει στον άξονα (anchor `center`, 🟢), foot = προβολή του cursor στον άξονα.
- ο cursor **πιο κοντά σε παρειά** (εξωτερική μισή ζώνη) ή **πέρα από τις άκρες** → πέφτει στο υπάρχον §3.7 **flush** (καμία αλλαγή).

**FULL SSoT — ΕΝΑ αρχείο, μηδέν νέο wiring/store/geometry, μηδέν διπλότυπο:**
- Επέκταση **μόνο** του `bim/columns/column-face-snap.ts`: για στόχους **τοίχου** (`endsAxis=null`) χτίζεται ένα `WallAxisFrame` (χορδή `axis[0]→axis[last]` + perp ημι-πάχος) μέσω **reuse `projectPolygonOnAxis`** SSoT· νέο `resolveWallAxisCenter` δοκιμάζεται **πρώτο** στο `resolveForTarget` (reuse **`projectPointOnAxis`** για foot+perp) και επιστρέφει anchor `center` / status `beam` / position στον άξονα — αλλιώς `null` → flush. **Μηδέν νέο projection primitive.**
- **Καμία αλλαγή σε scheduler/commit/store:** το αποτέλεσμα ρέει αυτούσιο μέσα από την **υπάρχουσα** §3.7 υποδομή — `snap-scheduler.applyColumnFaceSnap` → `setColumnFaceAnchor('center')` + `setColumnGhostStatus('beam')` + `setImmediateSnap(axis foot)`· `mouse-handler-up` → ίδιο resolver· `commitColumnFromState` anchor precedence (`faceAnchor='center'` νικά) → **preview ≡ commit** byte-for-byte. Ευθύς τοίχος → χορδή ≡ άξονας (ακριβής foot)· καμπύλος → χορδή προσέγγιση (συνεπές με `buildTargetFrame`).

### 3.10 Sync-in-preview unification — μία snap pipeline για όλα τα placement tools (2026-06-22)

**Giorgio order (Revit-grade, FULL ENTERPRISE + FULL SSoT):** δύο εργαλεία τοποθέτησης με **θεμελιωδώς διαφορετική αρχιτεκτονική face-snap** = anti-pattern. Κάθε στόχος (κολώνα/δοκάρι/τοίχος/**ακμή πλάκας**/μελλοντικός) δουλεύει **αυτόματα παντού** → εξαφανίζεται ολόκληρη η κλάση bugs «δουλεύει στο Α, όχι στο Β».

**FULL SSoT — reuse υπάρχοντα, μηδέν διπλότυπο:**
- **NEW** `bim/columns/column-preview-store.ts` — zero-React store με pre-collected στόχους (`footprints`/`beamTargets`/`wallTargets`/`slabTargets`), mirror του `wall-preview-store`/`beam-preview-store` (targets μέρος). Pre-collected (ΟΧΙ per-frame) → ελαφρύ hover.

- ✅ §3.9: η κολώνα κεντράρει στον **άξονα του τοίχου** (κέντρο ≡ άξονας) όταν ο cursor είναι κοντά στο centerline· οι παρειές παραμένουν προσβάσιμες (flush) στην εξωτερική ζώνη — μηδέν regression στο §3.7.
