## 2. Τι είναι το Nestor App

Πολυχρηστική, πολυεταιρική (multi-tenant) web εφαρμογή για **διαχείριση κατασκευαστικών έργων** και των οντοτήτων που σχετίζονται με αυτά.

**Τεχνολογία (συνοπτικά)**:
- Next.js 15 (App Router) + React + TypeScript
- Firebase: Auth, Firestore, Storage, Functions
- Hosting παραγωγής: Netcup (`nestorconstruct.gr`). Vercel χρησιμοποιείται για preview deployments.
- Πληρέστερη τεχνική τεκμηρίωση: `docs/centralized-systems/README.md` και ADR index.

**Παραγωγικό περιβάλλον**: `https://nestorconstruct.gr` (και `https://nestor-app.vercel.app` για preview).

## 3. Η διπλή υπόσταση του ιδιοκτήτη μέσα στην εφαρμογή

Ο Γιώργος εμφανίζεται **δύο φορές** στο Nestor App, με **εντελώς διαφορετικό ρόλο** σε κάθε περίπτωση:

### 3.1 Ως **χρήστης** (Firebase Auth user)

- **Τι είναι**: Ένας λογαριασμός εισόδου στην εφαρμογή.
- **Πώς αναγνωρίζεται**: Από το Gmail `georgios.pagonis@gmail.com` (τρέχον) + Firebase UID.
- **Δικαιώματα**: `globalRole = super_admin` → δικαίωμα να βλέπει **όλες** τις εταιρείες του συστήματος.
- **Σκοπός**: Καθορίζει **"ποιος μπαίνει"** στην εφαρμογή.

### 3.2 Ως **επαφή / φυσικό πρόσωπο** (Firestore contact, type='person')

- **Τι είναι**: Μία εγγραφή στη βάση δεδομένων που αναπαριστά τον Γιώργο **ως άνθρωπο** μέσα στα δεδομένα.
- **Πώς αναγνωρίζεται**: Από ξεχωριστό contact ID στο Firestore.
- **Σκοπός**: Συνδέεται με έργα και ρόλους — π.χ. "αρχιτέκτων στο έργο Χ", "επιβλέπων στο έργο Υ".
- **Είναι ανεξάρτητο** από τον λογαριασμό εισόδου.

### Γιατί αυτή η διάκριση

Επειδή ο ίδιος άνθρωπος μπορεί να έχει **πολλούς επαγγελματικούς ρόλους** σε **πολλά έργα πολλών εταιρειών**, ενώ ο λογαριασμός εισόδου είναι **ένας και μοναδικός**. Παράδειγμα από άλλο domain: ένας δικηγόρος έχει **έναν** λογαριασμό στο σύστημα της εταιρείας του, αλλά εμφανίζεται **ως αντίδικος, ως μάρτυρας, ως δικηγόρος** σε διαφορετικά dossiers.

## 4. Ιεραρχία οντοτήτων μέσα στην εφαρμογή

### 4.1 Επαφές (Contacts)

Όλες οι "επαφές" αποθηκεύονται στη συλλογή `contacts` του Firestore και χωρίζονται σε:

| Τύπος (`type`) | Παραδείγματα |
|----------------|--------------|
| `company` (νομικό πρόσωπο) | ΠΑΓΩΝΗΣ ΕΝΕΡΓΕΙΑΚΗ ΚΑΤΑΣΚΕΥΑΣΤΙΚΗ Α.Ε., TechStart Solutions, εργολάβοι, προμηθευτές |
| `person` (φυσικό πρόσωπο) | Πελάτες, ιδιοκτήτες, μηχανικοί, εργαζόμενοι — **και ο ίδιος ο Γιώργος** |
| (Δημόσιες υπηρεσίες) | Δ.Ο.Υ., Πολεοδομία, ΔΕΗ, ΕΥΔΑΠ — αντιμετωπίζονται ως ειδική κατηγορία επαφών |

### 4.2 Διαχωρισμός εταιρειών σε δύο "ρόλους"

Δεν αναλαμβάνουν όλα τα νομικά πρόσωπα έργα:

- **Εταιρείες που αναλαμβάνουν έργα** — συνδέονται με ένα `project` μέσω του πεδίου `linkedCompanyId`. Παράδειγμα: ΠΑΓΩΝΗΣ ΕΝΕΡΓΕΙΑΚΗ.
- **Εταιρείες-συνεργάτες** — υπάρχουν στις επαφές αλλά δεν "κατέχουν" έργα. Παράδειγμα: ένας προμηθευτής υλικών.

### 4.3 Δένδρο έργου

Κάθε **έργο** που έχει αναλάβει μια εταιρεία ξεδιπλώνεται ιεραρχικά:

```
ΕΤΑΙΡΕΙΑ (νομικό πρόσωπο που αναλαμβάνει)
   │
   └── ΕΡΓΟ
          │
          └── ΚΤΙΡΙΟ
                 │
                 └── ΟΡΟΦΟΣ
                        │
                        └── ΑΚΙΝΗΤΟ (διαμέρισμα, κατάστημα κ.λπ.)
                               │
                               └── ΠΑΡΑΚΟΛΟΥΘΗΜΑΤΑ
                                      ├── Θέση στάθμευσης
                                      └── Αποθήκη
```

### 4.4 Ρόλοι έργου (project members)

Συνδέουν **φυσικό πρόσωπο** (από τις επαφές) με ένα **έργο**, σε συγκεκριμένο ρόλο:

| Παράδειγμα | Φυσικό πρόσωπο | Ρόλος στο έργο | Έργο |
|-----------|----------------|----------------|------|
| 1 | Γιώργος Παγώνης | Αρχιτέκτων | Πολυκατοικία Α |
| 2 | Γιώργος Παγώνης | Επιβλέπων | Μονοκατοικία Β |
| 3 | Άλλο πρόσωπο | Πολιτικός μηχανικός | Πολυκατοικία Α |

Έτσι, **ο ίδιος άνθρωπος** (Γιώργος) εμφανίζεται με **δύο διαφορετικούς ρόλους σε δύο διαφορετικά έργα** — και αυτό μοντελοποιείται καθαρά μέσω του pattern επαφή ↔ ρόλος έργου.

## 6. Συχνά σημεία σύγχυσης

| Σύγχυση | Διευκρίνιση |
|---------|-------------|
| "Pagonis TEK" είναι εταιρεία | **Όχι** — είναι απλώς ετικέτα/όνομα εμφάνισης του Gmail λογαριασμού εισόδου. Δεν αντιστοιχεί σε καμία λειτουργική επιχείρηση μέσα στο σύστημα. |
| Ο super_admin user "είναι" η ΠΑΓΩΝΗΣ ΕΝΕΡΓΕΙΑΚΗ | **Όχι** — ο user είναι ο **άνθρωπος** Γιώργος. Η ΠΑΓΩΝΗΣ ΕΝΕΡΓΕΙΑΚΗ είναι **νομικό πρόσωπο μέσα στις επαφές** που "ανήκει" στον tenant που έχει επιλέξει από τον switcher. |
| Όλες οι εταιρείες στις επαφές = όλες οι εταιρείες-tenants | **Όχι** — υπάρχουν **νομικά πρόσωπα στις επαφές** (συνεργάτες, πελάτες) και υπάρχουν **εταιρείες-tenants** (που είναι τα "διαμερίσματα" του multi-tenant συστήματος). Είναι διαφορετικά πράγματα. |
| Το switcher αλλάζει χρήστη | **Όχι** — αλλάζει **εταιρεία προβολής**. Ο χρήστης παραμένει ο ίδιος. Μόνο οι super_admins έχουν πρόσβαση στον switcher. |
