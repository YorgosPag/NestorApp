## ΜΕΡΟΣ 1 — GROUND TRUTH Phase B (διάβασέ τα ΠΡΙΝ κώδικα)

### 🔴 Κρίσιμα ευρήματα (κώδικας = αλήθεια)
1. **Ο server ΠΕΤΑΕΙ το `kind`.** `src/app/api/floors/floors.handlers.ts` `handleCreateFloor` (~γρ. 151-159): `entitySpecificFields` περνά ΜΟΝΟ `number/name/buildingId/units/elevation/height`. Το `kind`/`longName`/`nameAutoGenerated`/`finishThickness` **χάνονται**, παρότι (α) η φόρμα τα στέλνει και (β) το `CreateFloorSchema` (floors.schemas.ts:25-30) **ήδη τα επικυρώνει** (`kind: FloorKindSchema` required). → **Pre-existing gap**: κανένας όροφος δεν persist-άρει `kind` σήμερα. Phase B πρέπει να τα threadάρει (boy-scout fix, N.0.2).
2. **Ο πίνακας δεν ξέρει το `kind`.** `useFloorsTabState.ts` `FloorRecord` (γρ. 32-42) **δεν έχει `kind`**· ο list handler δεν το επιστρέφει. → πρέπει να προστεθεί ώστε ο πίνακας να (α) δείχνει special rows διακριτά, (β) μετράει με `countBuildingStoreys`.
3. **Building PATCH:** `building-services.ts` `BuildingUpdatePayload` (γρ. 51-53) έχει `hasFoundation`/`foundationDepth`. Mirror → πρόσθεσε `hasStairPenthouse`/`stairPenthouseHeight`. Ο PATCH handler επικυρώνει με `BuildingElevationPatchSchema` (ΗΔΗ επεκτάθηκε Phase A) — επιβεβαίωσε ότι περνούν.
4. **Φόρμα:** `BuildingVerticalSetupForm.tsx` σήμερα (α) δεν περνά `kind` στο `createFloor` (το infer-άρει από number → **θα λάθος-ταξινομήσει foundation/penthouse**), (β) persist-άρει μόνο `hasFoundation`/`foundationDepth` building-level. Δεν δημιουργεί foundation/penthouse floor records.

### SSoT principles (ΑΠΑΡΑΒΑΤΑ)
- Το «Όροφοι: N» = **παράγωγο** μέσω `countBuildingStoreys` (REUSE Phase A)· ΟΧΙ νέο counter.
- `kind` = ΕΝΑ SSoT (`FloorKind`)· persist-άρεται end-to-end (φόρμα→schema→handler→Firestore→list→table).
- Foundation/penthouse = **και** building-level flags (datum truth) **και** floor records (drawable level). Μην διπλασιάσεις τη γεωμετρία — ο τύπος θεμελίωσης παραμένει per-element DXF (ADR-436/441).
- N.2 no `any`· N.7.1 αρχεία ≤500/functions ≤40· N.6 Firestore IDs μέσω enterprise-id (ο createEntity path το χειρίζεται ήδη).
