Η **μηχανολογική μελέτη θέρμανσης** (ΤΟΤΕΕ 20701-1, ΚΕΝΑΚ, EN 12831) χτίζεται σε στρώματα. **Ολοκληρωμένα (code = source of truth):**

| Στρώμα | Τι κάνει | Κατάσταση |

| **L1.5** | Πληρότητα φορτίου EN 12831: **Θερμογέφυρες (ΔU_TB) + Reheat (Φ_RH)** | ✅ (2026-06-09) |
| **L2** | Radiator sizing (EN 442) + boiler equipment-sizing | ✅ |

| **L4** | Hydraulic balancing (Darcy index-circuit + kv valves) + overlay | ✅ engine+viz |
| **L5** | Μηχανολογική Μελέτη — Printout/PDF Report (σύνοψη + 4 πίνακες) | ✅ |

**Τι έκλεισε μόλις (L1.5, για context):** ο engine υπολόγιζε μόνο `Φ_T+Φ_V`· τώρα ολοκληρωμένος `Φ_HL=Φ_T(+TB)+Φ_V+Φ_RH`. Θερμογέφυρες = blanket `U_corr=U+ΔU_TB` (αδιαφανή προς εξωτ./έδαφος)· reheat = `A_floor·f_RH`. Config presets στο `heat-load-config.ts`· per-space override `ThermalSpaceParams.thermalBridgeLevel?/reheatMode?` (ΜΗΔΕΝ Firestore schema)· additive result `{thermalBridgeW, reheatW}`· zero-regression default 0· 66/66 tests. ΕΚΤΟΣ ADR-040.

## 1) ΤΙ ΘΑ ΚΑΝΕΙΣ (L6)

Σήμερα η μελέτη υπολογίζει **φορτίο** αλλά **δεν ελέγχει κανονιστική συμμόρφωση κελύφους**. Οι «μεγάλοι παίχτες» (Revit Energy Analysis, 4M-FineHEAT/KENAK) μετά τον υπολογισμό φορτίου κάνουν **έλεγχο μέγιστου συντελεστή θερμοπερατότητας U** ανά δομικό στοιχείο έναντι των ορίων ΚΕΝΑΚ / ΤΟΤΕΕ 20701-1 ανά κλιματική ζώνη.

**Υπάρχει ήδη το θεμέλιο αλλά μισό:** `bim/thermal/kenak-thermal-config.ts` έχει `KENAK_MAX_U_WALL` (όριο U **μόνο εξωτ. τοίχου** ανά ζώνη A/B/C/D) + `isAboveKenakUMax(u, zone)` + `getKenakMaxUWall(zone)`. **Λείπουν τα όρια για δάπεδο/οροφή/στέγη/κούφωμα** και **ο per-boundary έλεγχος + το reporting**.

**Στόχος L6:** πλήρης πίνακας `U_max` ανά **στοιχείο × ζώνη** (ΤΟΤΕΕ 20701-1 Πίνακας 3.3α) + καθαρός engine που ελέγχει κάθε `HeatLoadBoundary` του χώρου (το resolver ΗΔΗ δίνει `kind`+`condition`+`uValue`) έναντι του ορίου → per-element ✓/✗ + νέα section «Έλεγχος Συμμόρφωσης ΚΕΝΑΚ» στο **L5 report** (δικό σου αρχείο). Όλα **derived**, μηδέν persist. **Advisory/soft** (όπως το υπάρχον ETICS pattern, ΔΕΝ μπλοκάρει).

**Αποτέλεσμα:** η μελέτη γίνεται **κανονιστικά πλήρης** — όχι μόνο «πόσο φορτίο» αλλά «περνάει ο ΚΕΝΑΚ;». Revit-grade energy-code check.

**ΠΡΩΤΟ ΒΗΜΑ: Plan Mode** — διάβασε kenak-thermal-config + resolver (boundaries) + L5 report, πάρε τις αποφάσεις (§2), ζήτα έγκριση.

## 2) ΑΠΟΦΑΣΕΙΣ ΝΑ ΚΛΕΙΔΩΣΕΙΣ ΣΤΟ PLAN MODE (πρότεινε + δικαιολόγησε)

- **D-A (πηγή ορίων U_max):** Πρότεινε **πλήρη Record `KENAK_MAX_U_*` ανά στοιχείο × ζώνη** (τοίχος ✓ υπάρχει· πρόσθεσε δάπεδο/οροφή-στέγη/ανοιγμα-κούφωμα) από ΤΟΤΕΕ 20701-1. Αντιπροσωπευτικές τιμές (editable, documented· κλώνος του υπάρχοντος `KENAK_MAX_U_WALL` pattern): π.χ. ζώνη Β τοίχος 0.45 / δάπεδο 0.45 / οροφή-στέγη 0.40 / κούφωμα 2.8 (επιβεβαίωσε με τον πίνακα). Πάρε ΕΣΥ τις τιμές ανά ζώνη — μην ρωτάς.
- **D-B (πού ζει το config — isolation):** Πρότεινε **νέο `bim/thermal/heat-load/kenak-envelope-limits.ts`** που κάνει `import type { ClimateZone }` + reuse το υπάρχον `KENAK_MAX_U_WALL` (ΟΧΙ duplicate τοίχου) και προσθέτει τα υπόλοιπα στοιχεία + `getKenakMaxU(kind, zone)` + `checkBoundaryCompliance(boundary, zone)`. **Γιατί νέο αρχείο:** το `kenak-thermal-config.ts` ανήκει στο ADR-396 (θερμοπρόσοψη ETICS) — additive επέκταση εκεί είναι ΟΚ αλλά νέο αρχείο στο heat-load/ κρατά το L6 SSoT μαζεμένο + isolation. Πάρε τη Revit-grade απόφαση (πρότεινε νέο αρχείο· δικαιολόγησε).
- **D-C (ποια στοιχεία ελέγχονται):** Πρότεινε έλεγχο **μόνο όσων αγγίζουν το όριο του κτιρίου** — `condition ∈ {external-air, ground}` (τα `adjacent-heated`/`unheated` δεν έχουν ΚΕΝΑΚ U_max εξωτ. κελύφους). Mapping `HeatLoadBoundaryKind → U_max key`: wall→wall, window/door→opening, floor→floor, roof→roof, ceiling→(εξαίρεσε ή roof). Πρόσεξε: το U του boundary είναι το **βασικό U** (το resolver δίνει `uValue` ΧΩΡΙΣ το ΔU_TB — το TB μπαίνει μόνο στον engine για φορτίο)· ο έλεγχος ΚΕΝΑΚ γίνεται στο **βασικό U** (κανονιστικά σωστό). Επιβεβαίωσε διαβάζοντας `space-boundary-resolver.ts`.
- **D-D (read-model):** Πρότεινε pure `deriveEnvelopeCompliance(spaces, inputs, zone)` που τρέχει το resolver (ΗΔΗ υπάρχει SSoT input gathering) και βγάζει per-element `{ kind, condition, uValue, uMax, compliant, refId }` + σύνοψη (πλήθος συμμορφούμενων/μη). **Reuse** το `useHeatLoadInputs` + `resolveSpaceBoundaries` (μηδέν νέα geometry). Η ζώνη υπάρχει ΗΔΗ στο `useHeatLoadInputs` (από `Building.climateZone`).
- **D-E (UI surface):** Πρότεινε **v1 = νέα section στο L5 report** «Έλεγχος Συμμόρφωσης ΚΕΝΑΚ» (πίνακας: χώρος/στοιχείο/U/U_max/συμμόρφωση ✓✗) — **δικό σου αρχείο `bim/thermal/report/*`, ΟΧΙ shared ribbon**. Προαιρετικό Slice B: overlay χρωματισμού μη-συμμορφούμενων στοιχείων (STAGE ADR-040, mirror HeatLoadOverlay) — ΑΛΛΑ απόφυγέ το αν αγγίζει shared view-tab toggles του boiler agent· κράτα το report-first.

## 5) ΜΟΝΑΔΕΣ + SSoT ΠΑΓΙΔΕΣ

- **Έλεγχος στο ΒΑΣΙΚΟ U** (`boundary.uValue`), ΟΧΙ στο `U_corr` (το ΔU_TB θερμογέφυρας είναι για το φορτίο, όχι για το κανονιστικό U_max). Διάβασε το resolver — το `uValue` είναι ήδη το καθαρό U.
- **Gate `external-air`/`ground`** — τα `adjacent-heated`/`unheated` ΔΕΝ ελέγχονται (δεν είναι εξωτ. κέλυφος).
- **Mapping kind→U_max key**: window+door → `opening` (κοινό όριο)· roof+ceiling → πρόσεξε (ceiling=adjacent συνήθως → skip). Πάρε τη Revit-grade απόφαση.
- **ΟΛΕΣ οι τιμές ορίων στο config** — μηδέν inline literal στον engine (όπως όλο το ADR-422).
- **Advisory/soft** — μην μπλοκάρεις τίποτα· είναι reporting (mirror ETICS `isBelowKenakAdvisory`).
- **i18n (report section):** keys ΠΡΩΤΑ σε `el` **ΚΑΙ** `en`· namespace `dxf-viewer-shell` (`thermalStudyReport.*`). Πρόσεξε τα **shared locale αρχεία** — μόνο δικά σου keys, additive.
- **ΜΗΝ persist** — pure derive (mirror L1-L5).
